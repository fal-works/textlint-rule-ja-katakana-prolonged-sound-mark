## 概要

外来語カタカナ語の末尾長音符（`ー`）の有無を統一する。

**Microsoft Japanese Localization Style Guide** をベースとしつつ、より自然な日本語を目指した独自拡張ルールによって判定する。

参考: [Microsoft - Localization Style Guides](https://learn.microsoft.com/en-us/globalization/reference/microsoft-style-guides)

## 検知方式

辞書方式（誤表記 → 正表記）による完全一致照合。辞書に収録されていない語は対象外。

## 基本規則

英語原語の語尾に基づき、カタカナ語末尾の長音符の有無を判定する。

下記の語尾パターンでは、長音符ありを原則とする:

- **子音 + -er/-or/-ar**: `ユーザー`（user）、`サーバー`（server）、`モニター`（monitor）
- **-y**（-ry, -ty, -phy を除く）: `ポリシー`（policy）、`テクノロジー`（technology）、`タクソノミー`（taxonomy）

下記の語尾パターンでは、長音符なしを原則とする:

- **母音 + -er/-or/-ar**: `クリア`（clear）、`ドア`（door）、`エンジニア` (engineer)
- **-ry**: `メモリ`（memory）、`ディレクトリ`（directory）、`ライブラリ`（library）
- **-ty**: `プロパティ`（property）、`セキュリティ`（security）
- **-phy**: `タイポグラフィ`（typography）
- **-ure**: `アーキテクチャ`（architecture）、`クロージャ`（closure）

## 慣用表記例外

上記原則に反する表記が慣用的に定着している場合は、例外として下記のように扱う:

- 原則では長音符ありでも、技術的な文脈に特有の語彙であるものは長音符なしとする。
    - 例: `コンパイラ`（compiler）
- 原則では長音符なしでも、技術的な文脈に特有の語彙でないものは長音符ありとする。
    - 例: `リカバリー`（recovery）
- 一般的な語彙でもありつつ技術文脈で特有の意味があるものや、固有名詞と衝突する可能性の高いものなどは、長音符あり・なし両方を許容する。
    - 例: `テクスチャー`/`テクスチャ`（texture）

## Microsoftスタイルガイドとの差異

Microsoftスタイルガイドの原則は以下の3段で構成される:

1. 英語語尾が -er/-or/-ar → 長音符あり
2. 上記以外で、長音符あり表記の実効文字数（※）が 4 未満 → 長音符あり
3. 上記以外 → 長音符なし

※ 実効文字数: 長音符は1文字として数えるが、小書き文字は文字数に数えない。

これに対し、本プロジェクトでは基本原則として以下の変更を加えている:

- **-er/-or/-ar の適用範囲を限定**: 母音 + -er/-or/-ar（-ear, -ier, -oor 等）を除外し、長音符なしとする。`クリア`（clear）、`ドア`（door）、`エンジニア` (engineer) など、該当する語の多くがMicrosoftスタイルガイドでも例外として列挙されており、同様の判断をより一般化したものと言える。
- **語尾パターンの細分化**: Microsoftスタイルガイドの原則では -er/-or/-ar 以外の長い語は「長音符なし」となるが、これを一律に適用すると不自然になる語も多く、そういった語は Microsoft の日本語ドキュメントでも表記揺れが見られる。実際に「長音符なし」で処理されることの多い語は -ry/-ty/-phy に偏っているため、本プロジェクトでは -ry/-ty/-phy 以外の -y 系（-cy/-gy/-my 等）は「長音符あり」を原則とすることで、より自然な表記を促すようにした。
- **文字数による判定を廃止**: 長音符あり表記で実効文字数4未満の語はそもそも数が少なく、実例を見ても誤用の危険が低いため、本プロジェクトでの管理対象外とした。

## 収録する語彙の選定基準

大まかには以下の基準で選定する:

- 情報系を主とした技術文書や技術記事、あるいはソースコードコメントなどで使われるカタカナ語を中心に収録する。
    - 備考: 情報システム系、ソフトウェア系などを中心としつつ、その他ハードウェア系なども積極的には除外しないものとする。
- 末尾長音符なしの表記が考えにくく誤用の危険が少ないと思われる語は、原則として収録対象外とする。
    - 例: `キー`（key）、`ペア`（pair）、`コピー`（copy）、`ロガー`（logger）、`メニュー`（menu）など、末尾長音を除いて2モーラ以下であるもの
    - 例: `トゥルー`（true）、`マーキー`（marquee）など、原語末尾の発音が長母音であるもの

## 適用範囲の制約

本プロジェクトの textlint rule は、テキスト中のカタカナ連続列を1トークンとして辞書と完全一致照合する。
そのため、複合語全体を辞書登録しない限り、複合語の先頭または途中に誤りがある場合は検知できない。

例: `サーバリスト`（`サーバーリスト` が正しいが、`サーバ` と `リスト` の境界をテキストから特定できないため未対応）

これは仕様上の制約として許容する。ただし、偽陰性の発生頻度が高いと思われるケースについては、複合語の辞書登録も検討する。
